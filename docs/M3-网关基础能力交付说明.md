# M3 网关基础能力交付说明

## 1. M3 目标

- 实现统一返回结构，避免接口响应风格不一致。
- 实现统一异常处理，避免异常直抛导致调用端无法稳定处理。
- 接入请求级 traceId，便于日志排查与链路关联。
- 给出可直接验证的示例接口（成功、参数异常、业务异常）。

## 2. 实际交付内容

## 2.1 统一返回与错误码

- `src/main/java/com/bones/gateway/common/ApiResponse.java`
- `src/main/java/com/bones/gateway/common/ErrorCode.java`
- `src/main/java/com/bones/gateway/common/BusinessException.java`

返回结构：
- `code`
- `message`
- `data`
- `traceId`
- `timestamp`

## 2.2 全局异常处理

- `src/main/java/com/bones/gateway/config/GlobalExceptionHandler.java`

已覆盖：
- `BusinessException`
- `MethodArgumentNotValidException`
- `ConstraintViolationException`
- `HttpMessageNotReadableException`
- `Exception` 兜底

## 2.3 traceId 过滤器

- `src/main/java/com/bones/gateway/config/TraceIdFilter.java`
- `src/main/java/com/bones/gateway/common/TraceIdContext.java`

行为：
- 优先读取请求头 `X-Trace-Id`
- 若不存在则自动生成
- 回写响应头 `X-Trace-Id`
- 注入 MDC，日志可打印 traceId

## 2.4 示例接口（验证入口）

- `src/main/java/com/bones/gateway/controller/DemoController.java`
- `src/main/java/com/bones/gateway/dto/EchoRequest.java`

接口：
- `GET /api/v1/demo/ping`：成功返回
- `POST /api/v1/demo/echo`：参数校验验证
- `GET /api/v1/demo/biz-error`：业务异常验证
- `GET /api/v1/demo/validate?number=0`：约束校验验证

## 2.5 配置补充

- `src/main/resources/application.yml`
  - 增加日志级别格式：包含 `traceId`

## 3. 验收结果

已完成以下验证：

1. 编译通过：
- `mvn -DskipTests compile`

2. 测试通过：
- `mvn test`

3. 运行后接口验证通过（端口 8083）：
- 成功返回：`/api/v1/demo/ping`
- 参数异常：`/api/v1/demo/echo`（空 content）
- 业务异常：`/api/v1/demo/biz-error`
- 约束异常：`/api/v1/demo/validate?number=0`

响应结果中 `code/message/traceId` 均符合预期，且响应头包含 `X-Trace-Id`。

## 4. 在 IDEA 中如何验证

1. 运行 `GatewayApplication`（`dev` profile，配置 `MYSQL_PASSWORD`）。
2. 打开 IDEA 内置 Terminal 执行：

```bash
curl -i http://localhost:8080/api/v1/demo/ping
curl -i -H "Content-Type: application/json" -d '{"content":""}' http://localhost:8080/api/v1/demo/echo
curl -i http://localhost:8080/api/v1/demo/biz-error
curl -i "http://localhost:8080/api/v1/demo/validate?number=0"
```

3. 观察点：
- 所有返回均为统一结构 `ApiResponse`
- 异常请求返回 `400`，且有明确业务码
- 响应头存在 `X-Trace-Id`
