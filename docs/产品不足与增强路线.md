# 产品不足与增强路线（截至 M8）

## 1. 当前不足（可立即改进）

## 1.1 身份与权限薄弱（已完成第一阶段）

现状：
- 已接入 JWT + RBAC，完成基础身份校验与权限隔离。
- 仍缺少组织/团队级权限模型（如 workspace/project/member）。

影响：
- 商业化场景不可用，审计与权限边界不足。

后续建议：
- 从“用户级权限”升级到“组织/项目级权限”。
- 增加细粒度策略（只读、管理、审计角色）。

## 1.2 会话标题与信息组织能力弱

现状：
- 会话默认标题“新会话”，缺少自动摘要命名与检索标签。

影响：
- 用户长期使用后难管理，开源演示吸引力一般。

建议：
- 首轮对话后自动生成标题（AI 摘要 12~20 字）。
- 增加标签与收藏能力。

## 1.3 上下文管理能力不足（已完成第一阶段）

现状：
- 已支持历史消息拼装与 Redis 缓存窗口。
- 仍缺少长对话自动摘要压缩与多层记忆策略。

影响：
- 对话连续性差，难支撑复杂任务。

后续建议：
- 引入“最近 N 条 + 摘要记忆”混合策略。
- 增加按模型上下文长度动态裁剪。

## 1.4 成本与配额控制（已完成基础版）

现状：
- 已支持按用户分钟级/日配额 + provider/model 日配额。
- 已支持日维度 token 与估算成本账本。
- 仍缺少“真实账单口径”（不同厂商 usage 字段精确对齐）。

影响：
- 商业模式与成本预测不稳定。

后续建议：
- 对接各厂商真实 usage 计费字段，替换估算口径。
- 增加成本告警与预算封顶策略。

## 1.5 可观测性仍偏底层（已完成部分增强）

现状：
- 已有 AI latency/error 指标，已新增首 token 时延（TTFT）。
- 仍缺少会话完成率、留存、失败重试率等产品化指标。

影响：
- 运营层难判断产品体验与商业健康度。

后续建议：
- 增加业务看板：TTFT、对话成功率、厂商故障率、fallback 命中率、用户留存。

## 2. 价值优先级建议（商业 + 开源）

1. 身份鉴权与配额体系（商业必需）
2. 上下文管理与缓存（体验与成本双提升）
3. 流式聊天落库与前端实时体验（演示效果显著）
4. 插件化 provider SPI 文档化（开源吸引力）
5. 运营指标看板（商业可运营）

## 3. 建议并入后续里程碑

- M9.1：组织级权限模型（workspace/member/role）与资源隔离
- M9.2：真实 usage 账本口径（按 provider/model/workspace 聚合）
- M9.3：运营看板指标与查询 API（成功率、fallback、TTFT、成本）

1. 本里程碑新增了哪些“用户可感知价值”？
2. 本里程碑降低了哪些“商业风险/运维风险”？
3. 下一里程碑优先补哪个能力最能提升留存或复用？
