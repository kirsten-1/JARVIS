## 3.1 AI Client 路由与 Fallback 切换

### 片段 A：路由 + provider fallback（同步/流式）

**位置**

- `src/main/java/com/bones/gateway/integration/ai/WebClientAiServiceClient.java:77-93`
- `src/main/java/com/bones/gateway/integration/ai/WebClientAiServiceClient.java:307-362`

```java
@Override
public Mono<AiChatResponse> chat(AiChatRequest request) {
    String traceId = resolveTraceId();
    ProviderContext primary = resolveProvider(resolveRequestedProvider(request), traceId);

    return executeChat(request, primary, traceId)
            .onErrorResume(error -> tryFallbackChat(request, primary, traceId, error));
}

private Mono<AiChatResponse> tryFallbackChat(AiChatRequest request,
                                              ProviderContext primary,
                                              String traceId,
                                              Throwable error) {
    ProviderContext fallback = resolveFallbackProvider(primary.name(), traceId, error);
    if (fallback == null) {
        return Mono.error(error);
    }
    opsMetricsService.recordProviderFallbackGlobal(primary.name(), fallback.name(), "sync");
    return executeChat(request, fallback, traceId);
}

private boolean isProviderFailure(Throwable throwable) {
    if (throwable instanceof BusinessException businessException) {
        ErrorCode code = businessException.getErrorCode();
        return code == ErrorCode.AI_SERVICE_TIMEOUT
                || code == ErrorCode.AI_SERVICE_UNAVAILABLE
                || code == ErrorCode.AI_SERVICE_BAD_RESPONSE;
    }
    return throwable instanceof TimeoutException || throwable instanceof WebClientRequestException;
}
```

**原理解析（高可用）**

- 不是“逢错就切”，只在 provider 故障语义上 fallback，避免业务异常被误切换。
- sync/stream 都复用相同策略，降低行为分叉。
- 通过 `opsMetricsService.recordProviderFallbackGlobal(...)` 记录 fallback 方向，方便后续治理。

**面试套话**

- 我在多模型兜底上做了“错误语义感知 fallback”，只对超时/不可达/坏响应切换，避免把业务错误当成基础设施错误。

## 6. 可直接背诵的“项目一句话”模板

1. 网关定位
- 我做的是一个统一 AI 网关，把多厂商协议差异、权限治理、计费和观测都收敛到了一个可复用入口。

2. 路由容灾
- 在多 provider 路由上，我做了错误语义感知 fallback，并补齐了 stream->sync 的降级通道，保证故障下仍可返回结果。

3. 安全隔离
- 权限上采用 JWT 无状态认证 + workspace 角色鉴权双层模型，确保身份可信和数据边界都可控。

4. 性能设计
- 缓存上采用上下文缓存、读缓存和热点响应缓存三层组合，写路径统一失效，兼顾性能和最终一致性。

5. 计费与观测
- 计费采用 actual/estimated 双口径，运营层同时输出成功率、fallback、TTFT 和成本趋势，支持日维度聚合分析。

## 8. 代码级风险与可优化点（面试加分项）

1. `ReadCacheService.evictByPrefix` 使用 `keys`，大规模 key 空间下可能阻塞 Redis。
建议：改 `SCAN` + 分批删除。

2. 流式 token 逐条 `updateMessageContent`，高 token 吞吐时 DB 压力偏高。
建议：按时间片/字符阈值批量 flush。

3. fallback 为单备 provider，策略维度较少。
建议：引入健康分 + 熔断半开 + 多备优选。

4. 计费目前在 Redis 聚合，长期账单建议异步落地明细表。
建议：增加 usage ledger 明细表 + 定时对账任务。
