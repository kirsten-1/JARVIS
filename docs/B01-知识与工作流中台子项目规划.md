# B01 知识与工作流中台子项目规划

## 项目定位

在 `gateway_pro` A03~A20 完成后，下一子项目进入 Jarvis 第二主线：

- **Knowledge Hub（RAG 中台）**
- **Workflow Hub（自动化编排中台）**

目标是把“可聊天”升级为“可持续生产价值”的个人 AI Infra：可管理知识、可自动执行任务、可观测可运营。

## 项目背景

你的业务场景是高频学习与研究，需要同时满足：

1. 多模型统一调用（已由 gateway/ai-service 打底）
2. 知识库可持续沉淀与高准确检索（RAG）
3. 自动化流程执行（类 n8n 的工作流编排）
4. 成本/稳定性可运营（观测、告警、限额、降级）

当前 A 阶段完成了“网关收官与证据闭环”，下一阶段要补齐“知识与流程生产力引擎”。

## 主要职责（下一子项目）

1. 设计并实现 RAG 中台：
   - 文档解析（PDF/Markdown/网页）
   - 分块策略（chunk/overlap 可配置）
   - 向量化与索引（优先 FAISS，后续可抽象 Milvus）
   - 混合检索（BM25 + 向量）与 rerank
   - 引用溯源与评测基线
2. 设计并实现 Workflow 中台：
   - DAG 节点模型（LLM/Tool/Condition/Loop/Approval/Webhook）
   - 执行引擎（拓扑执行、重试、超时、幂等）
   - 人工审核与回调机制
   - 模板化场景（代码审查、日报、摘要、邮件）
3. 打通网关集成：
   - gateway 侧新增知识检索与工作流触发 API
   - Agent 与 RAG/Workflow 协同（Supervisor 模式）
4. 建立运营与稳定性基线：
   - 任务成功率、平均耗时、失败类型、成本/Token 消耗
   - 告警规则与阶段验收报告机制（沿用 A 阶段门禁风格）

## 目标成果（量化口径）

1. RAG 质量目标：
   - QA 准确率从当前基线提升到 **>= 85%**（阶段目标）
   - 检索召回率 **>= 90%**
2. Workflow 自动化目标：
   - 覆盖 **>= 10** 个自动化场景
   - 流程成功率 **>= 95%**
3. 运行与成本目标：
   - 关键接口 P99 `< 1s`（网关侧）
   - 通过本地模型+云模型混合调度继续压低月成本

## B 阶段里程碑（建议）

1. **B01**：子项目骨架与技术选型冻结（本规划文档）
2. **B02**：文档解析 + 分块 + 入库管线 MVP
3. **B03**：向量检索 + BM25 混检 + rerank MVP
4. **B04**：RAG 评测集与离线评测脚本
5. **B05**：Workflow DAG 数据模型与执行引擎 MVP
6. **B06**：人工审核/Webhook/模板工作流首批
7. **B07**：gateway/agent 对接与端到端链路打通
8. **B08**：可观测与运维门禁（质量/性能/成本）收官

## 本周可执行清单（直接开工）

1. 在 `jarvis-platform` 下建立 `knowledge-workflow-service` 目录骨架。
2. 确定存储策略：
   - 元数据：MySQL
   - 索引：FAISS（本地）
   - 缓存：Redis
3. 输出 B02 的接口契约（ingest/search/retrieve/evaluate）。
4. 给 gateway 增加预留 API（feature flag 控制，不影响现网）。

## 风险与控制

1. **质量风险**：数据集不足导致评测波动  
   控制：先定义固定评测集与版本化评测脚本。
2. **复杂度风险**：Workflow 过早复杂化  
   控制：先做 MVP 节点集，再扩展循环/人工审核。
3. **成本风险**：检索与重排叠加成本上升  
   控制：分层召回（粗排→精排），对 rerank 设置阈值与采样。
